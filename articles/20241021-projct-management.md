fumiyaki です．

株式会社 tacoms で 5 月からお世話になっています．
エントリーブログ書きたいと思っていたら、プロジェクト管理を Notion に移行していたので、今回はその話を書きたいと思います．

## 当時の管理方法

私が入社した時は以下のツールを使用していました．

- 仕様管理: Notion
- プロジェクト管理: Asana
- コード管理: Github

当時はこれらのツールで管理していたのですが、課題がありました．

### 仕様とプロジェクトの管理の課題

ロードマップとタスクが同じプロジェクト単位で管理されていた
プロジェクト・ユーザーストーリーそれぞれで説明がどこに書いたらよいかが難しかった（Asana のロードマップのまとまりのやつの説明が書けない）
同じストーリーが Asana と Notion に存在していた
PdM が Notion でユーザーストーリ書いて，エンジニアが Asana に移動する運用
仕様は Wiki に書かれて，タスクは Asana に書かれていました．

エンジニアは Asana を見てタスクを確認するのですが，Asana にはタスク名と Notion への URL のみが記載されているため，仕様を確認するには Notion にアクセスする必要がありました．
タスクも階層があり，辿っていく必要がありました．
Notion に書かれた仕様の階層が辛く、情報を繋ぐための中間 DB があった（？）
PRD（仕様）とユーザーストーリーは別で管理されていた
PRDはリスト管理されていて構造化されていなかった
ユーザーストーリは構造化されていたが，PRD は構造化されていなかった

### プロジェクトとコードの管理の課題

Asana での管理
CamelOrder RoadMap
CamelOrder TaskBoard を使って管理していた
タスクを切ったら毎回タスクに依存関係や関連プロジェクトに紐づけする必要があった
しかもプロジェクトの XX という 2 段階で設定する必要があった
エンジニア側のタスク管理として，タスク・子タスクで管理したいと考えていました．
社内でやりたいタスク管理の方法が Asana と Github の連携が思ったようにできない

## 理想のプロジェクト管理

仕様・プロジェクト管理・コード全てが 1 箇所で管理されている

## 検討したこと

github project
notion

github project は有力な候補だったが管理しているリポジトリが複数あり，チーム単位でのプロジェクト管理が難しい

## 今

Notion のテンプレートを使用して仕様、プロジェクトの管理
Github との連携もやりたかったことが出来ている
